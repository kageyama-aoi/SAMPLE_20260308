# スキルの内容
--- name: github-issue-dev description: | GitHubのIssueテンプレートを活用した開発ワークフロースキル。修正依頼・追加開発・バグ報告など、ユーザーから何らかの開発作業を依頼された場合は必ずこのスキルを使うこと。 以下のような依頼があったら必ずこのスキルを使用すること： - バグ修正・不具合対応 - 機能追加・改修依頼 - リファクタリング・コード整理 - 「〜を直して」「〜を追加して」「〜を修正して」などの開発作業全般 - 開発後のgit add / commit作業 ワークフロー: Issueテンプレート確認 → Issue登録 → 開発 → Issue クローズ → git commit（候補提示） --- # GitHub Issue 駆動開発スキル このスキルは、開発作業を依頼されたときに **Issue登録 → 開発 → クローズ → コミット** の一連のワークフローを標準化するためのものです。 --- ## ワークフロー概要
1. Issueテンプレート確認
2. Issueを登録（テンプレートベース）
3. 登録済みIssue一覧をユーザーに展開
4. 開発作業を実施
5. Issueをクローズ
6. git statusでコミットメッセージ候補を提示・確認
7. ユーザー選択のメッセージでコミット
--- ## Step 1: Issueテンプレートの確認 まず .github/ISSUE_TEMPLATE/ ディレクトリにテンプレートが存在するか確認する。
bash
ls .github/ISSUE_TEMPLATE/
### テンプレートが存在しない場合 ユーザーに以下の旨を伝える： > ⚠️ .github/ISSUE_TEMPLATE/ にIssueテンプレートが見つかりませんでした。 > Issueテンプレートを設定することで、より構造化されたIssue管理が可能になります。 > テンプレートなしで続行することもできますが、設定を推奨します。 > 続行しますか？ 続行の場合はデフォルトのフォーマットでIssueを作成する（Step 2参照）。 ### テンプレートが存在する場合 テンプレートの内容を読み込み、依頼内容に合ったテンプレートを選択する。
bash
cat .github/ISSUE_TEMPLATE/<テンプレートファイル名>
--- ## Step 2: Issueの登録 ### GH CLIを使用する場合（推奨）
bash
# テンプレートを指定してインタラクティブに作成する場合
gh issue create --template "<テンプレートファイル名>"

# タイトルと本文を直接指定する場合
gh issue create \
  --title "<タイトル>" \
  --body "<本文（テンプレートに沿った内容）>" \
  --label "<ラベル>"
### Issueの本文フォーマット（テンプレートがない場合のデフォルト）
markdown
## 概要
<依頼内容の簡潔な説明>

## 詳細
<具体的な問題・要件>

## 期待する動作
<修正後・実装後にどうなるべきか>

## 補足
<再現手順、スクリーンショット、関連Issueなど>
--- ## Step 3: 登録済みIssue一覧の提示 Issue登録後、現在のオープンIssue一覧をユーザーに展開する。
bash
gh issue list --state open
出力例をユーザーにそのまま見せる（タイトルのみで可）。 > 📋 現在のオープンIssue一覧： > - #12 ログイン画面のバリデーションエラーを修正 > - #11 ダッシュボードのパフォーマンス改善 > - #10 （今回登録）〇〇の修正 --- ## Step 4: 開発作業の実施 ユーザーがIssue一覧を確認したら、該当Issueの内容に従って開発を進める。 - Issueの番号を念頭に置きながら作業する - 必要であれば作業中の進捗をユーザーに報告する --- ## Step 5: IssueのClose 開発完了後、該当Issueをクローズする。
bash
gh issue close <issue番号> --comment "実装完了。<簡潔な完了コメント>"
--- ## Step 6: コミットメッセージの候補提示 ### git statusで変更内容を確認
bash
git status
git diff --stat
### コミットメッセージ候補を3〜4つ生成 変更内容に基づき、以下のフォーマットでメッセージ候補を作成し、**選択形式**でユーザーに提示する。 **コミットメッセージフォーマット（Conventional Commits準拠）：**
<type>(<scope>): <summary>

Types: fix / feat / refactor / docs / style / test / chore
**ユーザーへの提示例：** > ✅ 開発が完了しました！コミットメッセージを選んでください： > > 1. fix(auth): ログインバリデーションのエラーメッセージを修正 > 2. fix: バリデーション失敗時のUI表示を修正 > 3. fix(login): 入力値検証ロジックのバグを修正 > 4. 自由入力（カスタムメッセージを入力） --- ## Step 7: コミットの実行 ユーザーが選択・確定したら、そのメッセージでコミットする。
bash
git add .
git commit -m "<選択されたコミットメッセージ>"
コミット後、結果を確認してユーザーに報告する：
bash
git log --oneline -3
--- ## 注意事項 - **必ずIssueテンプレートを確認してから**Issueを登録すること。テンプレートがない場合はユーザーに警告する。 - GH CLIが使えない環境では、GitHub APIやWebインターフェースを案内する。 - コミットメッセージはgit statusの実際の変更内容を見て生成すること（推測で作らない）。 - git add . の前に git status で変更ファイルを確認し、不要なファイルが含まれていないかチェックする。 - Issue番号はコミットメッセージに含めることを推奨する（例：fix(auth): ログインバグを修正 #12）。
